01. Architektura i przeplyw danych

Opis sposobu komunikacji frontend-backend-baza danych-Spotify API.

Model dzialania systemu

  1. Frontend inicjuje zapytanie HTTP do backendu.
  2. Backend wykonuje middleware i logike endpointu.
  3. W razie potrzeby backend komunikuje sie z PostgreSQL lub Spotify API.
  4. Odpowiedz JSON wraca do frontendu.
  5. Frontend renderuje rezultat i aktualizuje stan aplikacji.

Sposoby pracy

  • OAuth Spotify: dane pobierane przez access token.
  • Demo: dane pobierane z lokalnego zbioru mockData.

02. Backend rdzen i middleware

Analiza plikow odpowiedzialnych za start serwera i kontrole sesji.

backend/src/index.js

Plik konfiguruje serwer Express i rejestruje route API.

app.use('/api/auth', authRoutes)
app.use('/api/user', userRoutes)
app.use('/api/reviews', reviewsRoutes)
app.use('/api/spotify', spotifyRoutes)
app.use('/api/playlists', playlistsRoutes)
app.use('/api/recommendations', recommendationsRoutes)

Plik zawiera rowniez endpoint health check oraz middleware globalnej obslugi bledow.

backend/src/middleware/auth.js

Middleware realizuje walidacje tokenu JWT i odswiezanie sesji.

const token = req.cookies.token || req.headers.authorization?.split(' ')[1]
const decoded = jwt.verify(token, process.env.JWT_SECRET)
req.userId = decoded.userId
req.spotifyAccessToken = decoded.spotifyAccessToken

Po wykryciu wygasniecia tokenu wykonywany jest proces refresh token przez Spotify.

backend/src/lib/prisma.js i backend/src/lib/spotify.js

  • prisma.js: singleton PrismaClient.
  • spotify.js: OAuth, wyszukiwanie, top tracks, audio features, playlist sync.

04. Baza danych i Prisma

Opis modeli danych oraz relacji stosowanych w projekcie.

Zrodlo konfiguracji

Plik: backend/prisma/schema.prisma

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

Modele

  • User: profil, preferencje, relacje do recenzji i playlist.
  • Review: ocena i opis utworu, dane audio, relacja do usera.
  • Playlist oraz PlaylistTrack: struktura playlist i kolejnosc utworow.
  • LikedTrack: zapis utworow oznaczonych przez uzytkownika.

Ograniczenia i spojnosc

  • unikalnosc recenzji user-track.
  • unikalnosc tracka w jednej playliscie.
  • relacje z kasowaniem kaskadowym.

06_frontend_strony_ui.html

06. Frontend strony i komponenty

Opis logiki ekranow i komponentow interfejsu.

Strony aplikacji

  • Home i Login: uruchomienie sesji Spotify lub demo.
  • Dashboard: top tracks, top artists, statystyki.
  • Search: wyszukiwanie utworow i prezentacja listy.
  • Track: szczegoly utworu, recenzje, audio features.
  • Reviews: przeglad i usuwanie recenzji usera.
  • PlaylistGenerator: generowanie playlisty po gatunku lub tracku.
  • Profile: dane profilu i preferencje gatunkowe.
  • Community: wyszukiwanie i podglad profili publicznych.

Komponenty wspolne

  • Layout: nawigacja i kontener widokow.
  • ProtectedRoute: kontrola dostepu do tras.
  • StarRating: obsluga oceny co 0.5.
  • AudioFeaturesDisplay: wizualizacja cech audio utworu.
  • TrackCard: karta prezentacji utworu.

07. Uruchomienie i analiza techniczna

Sekcja zawiera konfiguracje startowa oraz opis mechanizmow stabilnosci dzialania.

Uruchomienie backend

cd backend
npm install
npm run db:generate
npm run db:push
npm run dev

Uruchomienie frontend

cd frontend
npm install
npm run dev

Konfiguracja backend .env

PORT=3001
CLIENT_URL=http://localhost:5173
DATABASE_URL=postgresql://USER:PASS@localhost:5432/waveeRating
JWT_SECRET=sekret
SPOTIFY_CLIENT_ID=...
SPOTIFY_CLIENT_SECRET=...
SPOTIFY_REDIRECT_URI=http://localhost:3001/api/auth/callback

Mechanizmy dzialania

  • Walidacja danych wejsciowych przez Zod.
  • Autoryzacja przez JWT i middleware auth.
  • Obsluga sesji z odswiezaniem tokenu.
  • Dane demo oparte o mockData przy braku tokenow Spotify.
